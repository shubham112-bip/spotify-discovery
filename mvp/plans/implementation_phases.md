# Commute Compass — Phase-wise Implementation Plan

This plan breaks down the development of the **Commute Compass MVP** into logical daily execution milestones.

---

## Phase 1: Authentication & Setup (Days 7–8)

### Day 7 · MVP Setup & Spotify OAuth
* **Goal**: Basic Express app scaffolding, `.env` config setup, and successful Spotify OAuth authorization.
* **Tasks**:
  - Initialize Node project inside `mvp/commute-compass/` with `npm init -y`.
  - Install dependencies: `express`, `dotenv`, `node-fetch`.
  - Create server skeleton (`mvp/commute-compass/server.js`) and environment config templates.
  - Create the landing page (`index.html`) with a "Connect Spotify" action.
  - Implement the Spotify Auth redirect and code exchange API endpoints.
  - **Success Criteria**: Opening `http://localhost:3001` and clicking "Connect Spotify" successfully logs in and returns user's profile info.

### Day 8 · Context Form & Frontend Mock
* **Goal**: Build the context questionnaire frontend interface and prepare production deployment staging.
* **Tasks**:
  - Implement form fields:
    * Commute length (15 / 30 / 45 / 60 mins)
    * Mood selection (Stressed / Calm / Energized / Tired)
    * Adventure level slider (1–5: Safe → Exploratory)
  - Create responsive Spotify-themed UI styles using CSS variables.
  - Set up deployment structure on **Vercel** or **Render** and configure redirect URLs.
  - **Success Criteria**: Submitting the form shows a loading state and successfully sends form criteria + auth token to the backend.

---

## Phase 2: Core Logic & AI Integration (Days 9–10)

### Day 9 · Gemini Session Plan Generator
* **Goal**: Call Spotify to retrieve user taste profile and feed it alongside form context to Gemini for plan generation.
* **Tasks**:
  - Write backend API route `POST /api/generate-session`.
  - Fetch user's top tracks and artists from Spotify API.
  - Design Gemini session generation prompt (forcing strict JSON output).
  - Parse Gemini response and display the 8 recommended tracks with explanations on the frontend.
  - **Success Criteria**: Submitting the form generates a JSON list of 8 tracks with tailored "why" cards explaining how they fit the user's commute context.

### Day 10 · Playlist Creation & Verification
* **Goal**: Convert Gemini's text recommendations into an interactive Spotify playlist.
* **Tasks**:
  - Implement Spotify Search logic to match track + artist text with active Spotify track URIs.
  - Create playlist API wrapper (`POST /users/{id}/playlists`).
  - Add matched tracks to the newly created playlist.
  - Display "Open in Spotify" button and render embed widget.
  - **Success Criteria**: Clicking the playlist button opens a playlist in the Spotify app containing the exact tracks generated by Gemini.

---

## Phase 3: Polish, Hardening & Deploy (Days 11–14)

### Day 11 · UX Polish & Synthesis
* **Goal**: Align findings from user interviews with MVP features, add loading indicators, and test error boundaries.
* **Tasks**:
  - Add visual loaders/skeletons during the 5–10 second LLM generation.
  - Handle exceptions (e.g. invalid tokens, track search misses).
  - Incorporate feedback from early user interviews (e.g. adjusting "why" card copy to highlight context over raw stats).
  - **Success Criteria**: Graceful handling of network timeouts or missing Spotify permissions.

### Day 12 · Problem Statement Lockdown (Part 3)
* **Goal**: Finalize problem definition research files, ensuring evidence from interviews supports the MVP's value proposition.
* **Tasks**:
  - Write `research/problem_statement.md`.
  - Map specific quotes from interview sheets to MVP features.
  - **Success Criteria**: Documented proof that commuters prefer "explained discovery" over generic recommendations.

### Day 13 · Production Hardening
* **Goal**: Configure environment variables on the production host and run incognito tests.
* **Tasks**:
  - Set up production `SPOTIFY_CLIENT_ID`, `SPOTIFY_CLIENT_SECRET`, and `GEMINI_API_KEY` on Vercel/Render.
  - Run the happy path demo 3 times from different user profiles.
  - **Success Criteria**: Flawless execution of the auth-to-playlist pipeline on the public staging URL.

### Day 14 · Production Release & Demo Video
* **Goal**: Promote the staging app to final production and record the project walkthrough.
* **Tasks**:
  - Deploy final stable build to the public URL.
  - Record a 2-minute Loom walkthrough showing: Form -> AI Plan with Explanations -> Playlist open in Spotify.
  - Update root `README.md` with active links.
  - **Success Criteria**: Entire project submission pack live and accessible via public links.
